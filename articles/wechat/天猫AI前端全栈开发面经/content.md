### Q1: 请简单介绍一下事件循环机制，特别是微任务和宏任务在浏览器中的执行顺序。

**难度**: ⭐
**考点**: JavaScript 异步执行机制

**答案要点**:

事件循环是 JavaScript 处理异步操作的核心机制，通过任务队列的协调确保单线程下的非阻塞运行。

- 调用栈清空后，优先处理所有微任务队列中的任务
- 微任务（Promise.then、MutationObserver）执行完毕后，浏览器可执行渲染
- 接着从宏任务队列（setTimeout, setInterval, I/O）中取出一个任务执行
- 如此循环往复，每个宏任务执行完都会检查并清空微任务队列

**常见坑**:
- 误认为 Promise 构造函数内部是异步的（实际上是同步执行）
- 混淆 Node.js 早期版本与浏览器环境事件循环的差异

**追问**:
- 如果在微任务中不断产生新的微任务，会发生什么？
- await 后面的代码属于什么任务？

### Q3: 请手写一个具有超时控制的 fetch 请求函数。

**难度**: ⭐⭐⭐
**考点**: Promise.all(), AbortController

**答案要点**:

利用 AbortController 接口可以实现对 fetch 请求的取消操作，结合 setTimeout 即可完成超时控制。

- 创建 AbortController 实例并获取其 signal
- 将 signal 传入 fetch 的配置项中
- 设置 setTimeout，在定时器触发时调用 controller.abort()
- 记得在请求成功后清除定时器，避免内存泄漏

**常见坑**:
- 在 catch 块中区分是业务错误还是超时导致的 AbortError
- 定时器没有及时清理

### Q5: TypeScript 中的 Type 和 Interface 有什么区别？在定义 AI 接口返回数据时你倾向用哪个？

**难度**: ⭐⭐
**考点**: TypeScript 基础

**答案要点**:

Interface 主要用于定义对象形状且支持声明合并，而 Type 更加灵活，支持联合类型和元组。

- Interface 可以通过重复声明自动合并属性，适合定义公共 SDK 接口
- Type 可以定义基本类型别名、联合类型（Union）和交叉类型
- 在定义 AI 响应时，由于响应结构可能包含多种状态（流式中、完成、错误），Type 的联合类型更有优势
- 性能上 Interface 略优，但在现代 TS 版本中差异极小

**常见坑**:
- 混淆了 extends 和交叉类型 (&) 的行为差异

**追问**:
- 什么是索引签名？在处理 AI 插件的动态参数时怎么用？

### Q7: 谈谈 React 中的 useEffect 闭包陷阱，以及在处理 AI 消息列表更新时如何避免？

**难度**: ⭐⭐⭐⭐
**考点**: React Hooks 原理

**答案要点**:

闭包陷阱是指 useEffect 的回调函数捕获了旧版本的 state 或 props，导致逻辑执行时数据不是最新的。

- 产生原因：依赖数组（dependency array）未正确包含所引用的变量
- 表现：在异步回调或定时器中读取到的 state 始终是初始值
- 解决：正确设置依赖项，或者使用 functional update（如 setState(prev => ...)）
- 在 AI 场景中，处理流式消息追加时，务必使用函数式更新以确保拿到最新的消息数组

**常见坑**:
- 盲目通过 useRef 绕过闭包陷阱，可能导致 UI 不触发重新渲染

### Q8: AI 聊天界面中，当消息内容很长且包含 Markdown 时，如何保证流式渲染的性能和流畅度？

**难度**: ⭐⭐⭐
**考点**: 长列表渲染、DOM 性能、Markdown 解析

**答案要点**:

流式渲染需要平衡解析频率和渲染开销，避免频繁的 DOM 重绘导致页面卡顿。

- 增量解析：不要每次收到新字符都重新解析整个字符串，可以记录已解析的位置
- 节流渲染：通过 requestAnimationFrame 或节流函数控制 UI 更新频率
- 虚拟列表：当对话历史过长时，引入虚拟滚动技术只渲染可视区内容
- 结构优化：将 Markdown 解析逻辑放在 Web Worker 中，避免阻塞主线程

**常见坑**:
- 每次追加字符都触发整个 Markdown 组件的重新渲染
- 滚动条自动触底逻辑与用户手动滚动冲突

**追问**:

### Q10: 请详细描述一下 AI Agent 的基本架构，前端在其中扮演什么角色？

**难度**: ⭐⭐⭐⭐
**考点**: AI Agent、系统设计

**答案要点**:

AI Agent 是能够自主规划、调用工具并完成任务的系统，前端主要负责交互展示和工具执行的反馈。

- 核心组件：规划（Planning）、记忆（Memory）、工具使用（Tool Use）
- 前端角色：作为"工具箱"的提供者，执行 Agent 发出的 Client-side Tools（如获取当前地理位置、操作 DOM）
- 状态追踪：展示 Agent 的 Thought、Action 和 Observation
- 交互干预：在 Agent 执行高危操作前提供人工确认（Human-in-the-loop）机制

**常见坑**:
- Agent 陷入死循环调用时，前端缺乏中断机制
- 思考过程展示过于简陋，用户感知不到进度

### Q13: 在 Node.js 中如何实现一个简单的转发层来处理 AI 请求？为什么要加这一层？

**难度**: ⭐⭐⭐
**考点**: BFF 层、安全、流式转发

**答案要点**:

在前端与 LLM API 之间增加 Node.js 层可以解决安全、跨域和协议转换问题。

- 安全性：隐藏 API Key，防止前端泄露
- 协议转换：将后端的复杂流式协议统一封装为前端易用的 SSE
- 缓存与限流：对相同问题的请求进行缓存，并控制单个用户的调用频率
- 内容审计：对输入输出进行合规性检查

**常见坑**:
- Node.js 层没有正确流式转发，而是等全部生成完才返回，导致首字延迟（TTFT）过高
- 内存泄漏，未正确销毁转发的流对象

### Q15: 如何评估一个 AI 前端应用的性能？有哪些关键指标？

**难度**: ⭐⭐⭐⭐
**考点**: 性能监控、AI 特有指标

**答案要点**:

除了传统的 Web 指标，AI 应用更关注交互的实时性和生成质量。

- **TTFT（Time to First Token）**: 从发送请求到看到第一个字的时间，直接影响首屏感知
- **TPS（Tokens Per Second）**: 生成速度，决定了阅读体验是否连贯
- **请求成功率**: LLM 接口由于负载问题常有波动，需监控重试率
- **用户反馈指标**: 点赞率、采纳率、手动修改率（反映生成质量）
- **资源占用**: 长对话场景下的内存增长情况

**常见坑**:
- 只关注页面加载速度（LCP），忽略了对话过程中的卡顿

### Q16: 如何实现一个支持多轮对话的消息管理系统？

**难度**: ⭐⭐⭐
**考点**: 状态管理、数据结构

**答案要点**:

多轮对话的核心是维护一个有序的消息数组，并正确处理历史上下文。

- 数据结构：每个消息包含 id, role（user/assistant/system）, content, timestamp 等
- 上下文裁剪：由于 Token 限制，需要策略性地只保留最近几轮对话或进行摘要压缩
- 持久化：利用 LocalStorage 或 IndexedDB 存储本地对话历史
- 状态同步：确保 UI 渲染与内存中的消息数组实时同步，处理好生成中的临时状态

**常见坑**:
- 消息 ID 重复导致渲染 Key 冲突
- 历史记录过多导致页面初始化加载缓慢

### Q17: 引入流式输出的 UX 优化策略

**答案要点**:

- 引入流式输出：首字出现速度比完整回复更重要
- 预加载与预测：根据用户输入习惯，提前预取可能的上下文数据
- Loading UI：展示 AI 的"思考步骤"或进度条，缓解焦虑
- 乐观更新：对于简单的指令，先更新 UI 状态，再等待后端确认

**常见坑**:
- 进度条设计不合理（卡在 99%）
- 忽略了移动端弱网环境下的超时处理

### Q18: 如果让你从零设计一个企业级的 AI Chat 组件库，你会考虑哪些核心模块？

**难度**: ⭐⭐⭐⭐
**考点**: 组件化设计、系统架构

**答案要点**:

一个成熟的 AI 组件库需要兼顾灵活性、扩展性和极致的交互体验。

- 核心渲染层：支持 Markdown、AST、数学公式、多媒体（图片/视频）的渲染引擎
- 状态管理中心：统一处理对话流、流式状态、历史记录和插件状态
- 适配器层：兼容不同大模型供应商（OpenAI, Anthropic, 内部模型）的 API 协议
- 交互增强：包含自动触底、手动滚动检测、复制、重试、反馈等内置功能
- 扩展插件系统：支持第三方工具集成和自定义 UI 片段

**常见坑**:
- 耦合太深，导致无法快速切换底层模型
- 忽略了无障碍（Accessibility）支持

### Q19: 在 AI 驱动的应用中，如何平衡"自动化"与"用户控制权"？

**难度**: ⭐⭐⭐⭐⭐
**考点**: 产品思维、AI UX

**答案要点**:

过度自动化会导致用户失去掌控感，而手动操作过多又失去 AI 的意义。

- 渐进式自动化：从建议模式开始，逐步过渡到自动执行
- 可逆操作：所有 AI 自动执行的步骤都应该可以撤回
- 透明度：让用户清楚知道 AI 正在做什么、为什么这样做
- 用户控制点：在关键决策节点保留人工确认
