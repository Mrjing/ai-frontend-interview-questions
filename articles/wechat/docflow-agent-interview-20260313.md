## Q1: 智能客服 Agent，多轮对话流程设计

假设你在美团，需要开发一个智能客服 Agent，来处理用户的退款、换货、催单等问题。面试官问：

**请设计一个多轮对话流程，处理用户"我要退款"的需求。**

关键点：

1. 对话状态管理：需要记录当前的"业务环节"（退款原因、退款金额、收货地址等）和"已填槽位"。
2. 每轮的解析：用户输入 → 意图识别 + 实体抽取 → 拿到这轮的结果。
3. 决策与动作：根据"状态+本轮意图"决定是追问、调接口、还是转人工。
4. 边界处理：超时、用户打断、多轮澄清。

一个简化版的状态机思路：

用户说"我要退款" → 系统识别意图：退款 → 系统查询：这个订单的退款状态 → 系统回复：好的，请问退款原因是？

后续：
用户说"商品有质量问题" → 系统记录原因 → 系统追问：请问退款金额是？
...
直到所有必要槽位填满 → 系统调用退款接口 → 反馈结果

但实际比这复杂得多，比如用户中途改口："算了不退了"或"我要换货"——需要处理好意图切换。

## Q2: 大模型幻觉问题

面试官问：**LLM 的幻觉问题你了解吗？在实际项目里怎么处理的？**

幻觉分几种：

1. 无中生有：模型编造不存在的信息（比如虚构论文、虚构API参数）
2. 张冠李戴：把A的信息安到B上（把别人的观点说成你的）
3. 过度肯定：对不确定的事情给出很确定的回答

应对策略：

- 检索增强（RAG）：让模型基于检索到的真实文档来回答，减少"自由发挥"的空间
- 约束输出格式：通过prompt明确告诉模型"如果不确定就回答不知道"，或者限定回答的范围
- 后处理校验：对模型输出做事实核查（交叉验证、规则校验）
- 用户反馈闭环：收集badcase，标注后做微调或者few-shot优化

## Q3: Agent 的规划能力（Planner）

面试官问：**Agent 的 Planner 模块你是怎么实现的？多任务协同怎么处理？**

Agent 要能"想清楚再动手"：

1. 任务分解（Decomposition）：把复杂任务拆成子任务序列
2. 工具选择（Tool Selection）：每一步选合适的工具
3. 执行与观察（Execute & Observe）：执行工具，拿到结果
4. 反思与调整（Reflection）：结果不对就调整计划

ReAct 模式是一个常用的思路：

- Thought：思考当前该做什么
- Action：选择一个工具并执行
- Observation：观察执行结果
- 重复直到完成

多任务协同：当有多个子任务时，可以用 Planner 先排好执行顺序和依赖关系，再逐个执行。步骤间的数据传递通过共享的 Memory 或上下文来实现。

## Q4: Agent 的工具调用

面试官问：**Function Calling 的技术方案你是怎么做的？**

工具调用的核心流程：

1. 工具描述（Tool Description）：给每个工具写清晰的描述，让模型知道什么时候该调哪个工具
2. 调用解析（Parse）：模型返回工具名+参数后，前端/后端解析并校验
3. 执行工具（Execute）：调用对应的函数/API
4. 结果回传（Return）：将执行结果返回给模型，让它继续推理

技术细节：
- 工具描述的写法很关键，描述不清会导致模型选错工具或传错参数
- 参数校验：在执行前对参数做校验（类型、范围、必填），避免执行出错
- 超时处理：工具调用可能很慢，需要设置超时时间
- 错误处理：工具执行失败时的回退策略（重试、换工具、报错给用户）
- 工具治理：随着工具数量增加，需要做工具分类、权限管理、使用频率统计

## Q5: Agent 效果评估与用户反馈

面试官问：**分享一次通过用户反馈优化 Agent 能力的经历。**

按"现象 → 归因 → 动作 → 效果"来组织回答：

1. **现象**：上线后发现某类问题用户常问但回答不准
2. **Badcase 收集**：从工单、日志、标注里筛出问题样本
3. **归因**：定位到是"意图、检索、生成、工具"哪一环出了问题
   - 比如：意图被误判 → 优化意图识别的few-shot
   - 检索不到对应文档 → 调整chunk策略或补充文档
   - 生成质量差 → 优化prompt或微调
   - 工具调用出错 → 优化工具描述或参数校验
4. **改进动作**：根据归因做针对性优化
5. **验证**：用A/B测试或离线评估验证改进效果

关键原则：
- 不要凭感觉改，要有数据支撑
- 建立badcase收集和分析的闭环流程
- 高频问题优先处理
