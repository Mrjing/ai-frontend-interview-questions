1. 介绍一下 SSE (Server-Sent Events) 和 WebSocket 的区别?
难度: 女女
考点: 网络协议、流式传输
SSE 是一种基于 HTTP 的单向通信机制，允许服务器向客户端推送数据，而 WebSocket 是全双工的通
信协议。
。 通信方向: SSE 是单向的 (Server 到 Client), WebSocket 是双向的
。 协议基础: SSE 基于标准 HTTP 协议，WebSocket 是独立的 TCP 协议
。 断线重连: SSE 内置了断线重连机制，WebSocket 需要手动实现
。 数据格式: SSE 默认传输文本/事件流，WebSocket 支持二进制和文本
常见坑:
。 误以为 SSE 只能传文本，其实可以通过 Base64 传二进制
。 忽略了浏览器对 SSE 并发连接数的限制 (Chrome 限制为 6 个)
追问:
。 为什么目前主流的 Al 对话 (如 ChatGPT) 多采用 SSE 而不是 WebSocket?

4. 说说 React Hooks 为什么不能写在条件判断里?
难度: 妇女
考点: React 原理、Fiber 架构
React Hooks 的调用顺序必须在每次泻染时保持一致，因为 React 内部是通过链表来按顺序存储和查找
Hook 状态的。
。 存储机制: 每个组件对应的 Fiber 节点上有一个 memoizedState 链表
。 索引匹配: React 依赖 Hook 调用的物理顺序来关联状态
。 破坏后果: 条件判断会导致链表偏移，导致后续 Hook 取到错误的状态
。 官方限制: eslint-plugin-react-hooks 插件会强制检查此规则

。 认为 Hook 是通过 key 来标识的 (实际上是靠顺序)
