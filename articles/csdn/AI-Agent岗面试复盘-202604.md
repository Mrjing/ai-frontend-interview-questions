## 一面：基础题，考察你真的懂 Agent 吗

### Q1：你做的 Agent 项目里，用了什么框架？为什么选它而不是其他？

用户回答基于LangChain。面试官追问LangChain的劣势及优化方向：

- LangChain重，抽象层级多，定制化改起来麻烦
- 很多场景不需要那么多组件，启动慢
- 优化方向：做分层架构，核心流程保留，组件可插拔

### Q2：Agent 最常见的失败场景是什么？你怎么解决？

三个面试官都问了。常见坑：

1. **工具调用失败**：LLM生成的参数不对、格式不对
   - 解法：参数校验层 + 失败重试 + 人工兜底
2. **上下文溢出**：对话轮数多，Context超了
   - 解法：上下文压缩 + summarize + sliding window
3. **目标漂移**：走着走着偏离了原始目标
   - 解法：每步做目标对齐 + 定期反思 + 必要时重新规划

### Q3：ReAct、CoT、ToT这些规划方法，实际效果怎么样？

用户实战感悟：
- CoT在复杂问题分解不好
- ReAct让模型边想边调用工具检索，准确率提升15%
- ToT效果好但token消耗多三倍，线上成本扛不住

关键点：说清楚trade-off，效果提升多少，成本增加多少。

## 二面：系统设计——设计面向企业客户的智能客服Agent

### 架构分层设计

```
┌─────────────┐
│ 接入层      │ ← 对接多渠道（网页、APP、公众号、企微）
├─────────────┤
│ 对话管理层  │ ← 上下文管理、多轮对话状态跟踪、意图识别
├─────────────┤
│ Agent 核心层 │ ← 规划、工具调用、反思、记忆
├─────────────┤
│ 工具层      │ ← 知识库检索、工单系统API、用户信息查询、物流查询
├─────────────┤
│ 输出管控层  │ ← 敏感词过滤、内容审核、话术规范
└─────────────┘
```

### 记忆设计
- 短期记忆：对话上下文，存Redis，设过期时间
- 长期记忆：用户画像、历史问题总结，存向量数据库，需要时召回

### 幻觉防控
1. 检索增强RAG：所有回答基于知识库内容
2. 置信度校验：没信心就转人工
3. 事实核查：输出与检索原文比对
4. 人工复核：关键场景必须过人工

### 稳定性保障
- 超时处理：设置超时时间，友好提示
- 降级策略：大模型挂了，降级到规则匹配或转人工
- 监控告警：跟踪每步成功率、失败原因

## 三面：深度发散

### Q：AI Agent最大的瓶颈是什么？

1. **可靠性**：复杂场景一步错步步错，很难保证99.9%稳定性
2. **成本**：一个复杂Agent调用好几次大模型，并发上去成本扛不住

### Q：从零搭建Agent团队怎么招人？

初期3-5人足够：
1. 一个有大模型应用经验的后端
2. 一个算法同学，做Prompt优化和召回优化
3. 一个产品，梳理场景和用户反馈
